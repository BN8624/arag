# golem — gemma 워커가 게임 룰엔진을 찍어내는 ARAG 서브프로젝트

## 한 줄
gemma(31B) 워커가 **JS 게임 룰엔진**을 select-best로 생성·자가수정하고 골든 오라클로 검증한다.
Claude는 계약·바인딩·조율만(머리), 워커는 구현만(손). ARAG 본체의 select-best·트레이스diff
철학을 JS로 옮긴 것이다.

## 왜 JS/웹인가
- gemma 강점 = Python·JS (공식 모델카드). 약점 = GDScript(저자원 언어, Godot3/4 혼동).
- 사용자는 폰에서 작업·판단한다 → 웹게임은 폰 브라우저에서 즉시 플레이·URL 공유가 된다.
- 타깃 장르(턴제 RPG·로그라이크·카드·시뮬·오토배틀러)는 로직 중심이라 룰엔진이 게임의 대부분이다.
- 근거·출처는 context-notes 결정 G1~G3.

## 역할 분담
- 워커(gemma) = JS 룰엔진 모듈 구현(손).
- Claude = 시스템 계약/아키텍처, 표현층 바인딩, 오케스트레이션(머리).
- 사용자 = 방향, 밸런스·손맛(오라클 없는 것).

## Phase 1 = 스케줄러 스파이크
가장 얽힌 부분(틱/속도게이지 턴 스케줄러 — ARAG가 실측한 frontier 벽)을 JS로 먼저 태운다.
뚫리면 나머지는 내리막, 안 뚫리면 한계를 싸게 배운다. 조각은 넷이다.
- `reference/` — 정답 스케줄러(레퍼런스 구현) + 골든 트레이스 생성기. 모델 비노출.
- Node 게이트 — 후보 JS 모듈을 node로 돌려 seed→트레이스 출력, 골든과 diff(콜0).
- 생성 프롬프트 — gemma에 "JS 한 모듈, 이 인터페이스"를 요구(stdlib Python 아님).
- 드라이버 — 생성→게이트→첫발산 힌트 자가수정→재시도(cap)→"cracked@N" 보고.

## 안 하는 것 (Phase 1)
표현층(Phaser/Canvas), 멀티 시스템 통합, 대시보드, 워커 11개 풀가동. 스케줄러 한 조각만 본다.
