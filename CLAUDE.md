# CLAUDE.md — 생성기 (Generator)

## 이 프로젝트가 뭔지

**생성기**다. 사용자가 아이디어 한 줄을 주면, gemma 4 모델 둘(26B·31B)이
설계→구현→검증→비평 루프를 자동으로 돌아 **멀티파일 Python 프로토타입**을 만들어내는 시스템.
루프가 한두 바퀴 돈 뒤 최종 결과물이 사용자에게 오고, 사용자가 피드백을 주면 다시 돈다.

> 주의: 이건 "개인툴"(폰에서 CLI 일꾼들을 부리는 웹 관제 도구)과 **다른 프로젝트**다.
> 생성기는 개인툴 위에서 운영될 대상이고, 이 저장소에는 생성기만 만든다.

## 방향 (2026-06-14 갱신)

- **정체성**: ARAG = **무료 gemma(26B/31B) × 오케스트레이션의 frontier**를 찾는다.
  상위/유료 모델 에스컬레이션은 **영구 폐기**(그들은 이미 그렇게 쓰임, novelty 0 —
  유일 예외: 무료모델 소멸 시). 끝점 = 무료모델 한계 지도, 또는 그 안의 쓸만한 생성기.
- **핵심 질문 둘**: ① 싼 모델을 오케스트레이션으로 어디까지 미나 ② 어느 역할(머리=설계·
  비평 / 손=구현)에 어느 모델이 맞나.
- **관측 우선 + 하네스 우선**: 실패를 보면 *모델 탓 전에 하네스/oracle 탓을 의심*하라.
  (이 프로젝트 반복 교훈 — "model_fail"의 상당수가 oracle/계약 버그였다.) 분류·점수는
  `observability.py`(limit_type, artifact_score, 콜0) + PLAN2 파생 레이어 `plan2.py`
  (라벨 5종 PASS/PARTIAL_USEFUL/MODEL_FAIL/INFRA_FAIL/HARNESS_FAIL).
- **측정의 핵심 원칙**: 설계 고정(`--resume`) + 같은 oracle + 한 번에 한 변수.
  변수 = 아키텍처(파일별 분해 vs `--whole` 통짜) × 모델(26B/31B) × cold/warm(노트) × 역할.
- **확정 결과(L2-3, 2026-06-14)**: 구현은 26B=31B / 분해=통짜(차이없음) / 노트 효과없음
  (이미 통과=천장). 차이는 더 어려운 카드에서만 갈림.
- **통합 frontier 규명(2026-06-15, 결정20·21·24)**: 난이도=창발적 통합(여러 시스템이 틱 단위로
  동시에 맞물려야). 조립카드 T-000012로 측정 → 단일시도 ≈1/3, select-best로 뚫림(설계 무관).
  frontier 정체 = **틱/속도게이지 턴 스케줄러 엣지케이스**(all-or-nothing). 다음 = warm 캠페인·형태확장.
- **Design Bank**는 카드 저장소로 재사용(`bank_*.py` + `design_bank.sqlite`). 설계자에
  Gemini 허용했었으나 현재는 무료 gemma만 사용.
- 명세·실험·아키텍처 사다리는 [PLAN.md](PLAN.md), 현재 상태는 [HANDOFF.md](HANDOFF.md),
  결정 로그는 [context-notes.md](context-notes.md).

## 사용자에 대해

- 사용자는 **바이브코더**다. 코드를 직접 읽고 쓰지 않는다.
- 결과물은 "되나/안 되나"와 사람 말 요약으로 판단한다.
- 거의 모든 작업을 폰에서 한다.

## 모델 / API (확정 사항)

- **구글 AI Studio API** 사용 (무료 플랜). Vertex 아님. 로컬 모델 아님.
- **생성기(generator) = gemma 4 26B** (MoE, 빠른 처리량 → 파일별 다회 생성에 적합)
- **비평가(critic)·설계자 = gemma 4 31B** (dense, 깊은 추론 → 설계와 판단에 적합)
- 쿼터(모델별 각각): **RPM 15, TPM 무제한, RPD 1,500, 단일 출력 32k 토큰(추론 포함)**
- 쿼터 대응 4원칙:
  1. **한 키 안에선 콜을 순차로**(키별 4초 페이서 = RPM 15 보호). 병렬은 **키를 늘려서** 한다 —
     워커=키, 시도 하나가 키 하나를 통째로 빌려 26·31 다 그 키로(결정22). 11키로 진짜 병렬.
     키×모델 RPD는 `key_usage.py`가 태평양시간 기준 집계·자동차단(결정23).
  2. **출력은 파일별로 쪼개 생성** (32k 출력 한도 보호. gemma 4의 thinking 토큰이
     출력 한도를 먹으므로, 프로젝트 전체를 한 응답에 뱉게 하지 말 것)
  3. 생성=26B / 비평·설계=31B 분담 (RPD가 모델별 별도라 서로 안 깎아먹음)
  4. **콜 래퍼에 최소 간격(4초) + 429 백오프 내장** (RPM 15 = 4초/콜이 안전선, 못 줄임.
     키별 페이서라 같은 키만 직렬, 다른 키는 독립. 429를 받으면 대기 후 재시도)
- 무료 티어는 입력이 모델 개선에 쓰일 수 있음 — 민감한 아이디어 입력 금지(README에 명시).

### 운영 구성 = 31solo 단독 (2026-06-15 갱신, 결정25)

측정·운영의 **기본 구성은 31solo(31B 단독)로 고정**한다. 헷갈리면 이걸 본다.

- **31solo = 머리·손 모두 31B.** 신뢰 베이스라인이자 운영 기본. select-best 루프·게이트·
  오케스트레이션은 단일 모델로 그대로 돈다.
- **두 모델 역할비교(머리/손 분리) arm은 보류.** 두 번째(손) 모델이 매일 반복되는 서버장애로
  사실상 못 쓰게 됨 → 사용자 결정으로 제외. **다시 필요하면 사용자가 먼저 꺼낸다.** 그 전엔
  비용·역할비교 측정을 제안·재개하지 말 것.
- 과거 두-모델 구성(결정16)의 폴백·역할분리 코드는 남아 있으나(31solo에선 자연 no-op),
  운영·로드맵에선 다루지 않는다.

## 산출물(프로토타입) 규격

- **언어: Python** (1차 고정)
- **형태: CLI/스크립트 도구** (1차. 가장 쉬우면서 멀티파일·실행 검증이 깔끔한 형태)
- **무조건 멀티파일.** 단일파일은 실패로 간주한다. (단일파일 생성은 이미 검증된 능력.
  이 프로젝트가 보려는 건 *파일 간 정합성* — import가 맞물리고 전체가 하나로 도는가.)
- 멀티파일 "위장" 금지: 파일만 여러 개고 서로 안 부르는 것, import만 하고 실제 사용
  안 하는 껍데기 모듈은 게이트에서 걸러낸다.
- **표준 라이브러리만** (1차). 외부 패키지를 허용하면 Docker 게이트가 의존성 설치·네트워크를
  떠안게 됨. 설계 단계에서 제약으로 박고, 정적 게이트에서 비-stdlib import를 걸러낸다.
  2차에서 화이트리스트로 완화 검토.
- **대화형 입력(stdin) 금지.** 입력은 CLI 인자/파일로만. `input()` 대기는 실행 게이트를
  영원히 멈추게 한다. 설계 규칙으로 박고, 실행 게이트에 건당 타임아웃을 둔다.

## 루프 구조 (확정)

```
아이디어 (사용자 입력)
  → [31B] 설계: 파일 구조 + 파일별 인터페이스 계약 + 의존관계 + 핵심사항 + 수용기준
  → [정적·콜0] 설계 검증: 파일 2개 이상? 순환참조 없나? 진입점 있나?
      실패 → 31B 재설계 (상한 있음)
  → [26B] 설계 계약대로 파일별 구현 (직전까지 만든 파일을 컨텍스트에 포함)
  → [정적·콜0] 게이트: ast 파싱 / import 해소·실사용 / 미정의 이름 / stub 탐지 / 진입점 존재
      / **설계 계약 대비 시그니처 정합성** (계약이 JSON에 있으므로 ast 대조는 공짜 — 모델 콜 불필요)
      / 비-stdlib import 탐지
      실패 → 26B 자가수정 (에러 위치·종류를 정확히 프롬프트에 넣어줌, 최대 K회)
  → [실행] Docker 게이트: 부팅, 즉시 크래시 없음, "성공 신호" 커맨드 1개 통과
      실패 → 26B 자가수정 (최대 K회)
  → [31B] 비평: 코드 + **증거**(실행 로그 + 정적검사 요약 + 수용기준 체크리스트) 기반
      비평은 **파일별 지적** 형태(구조화)로 받는다 — 전체 재생성이 아니라 표적 수정을 위해
      "더 나아질 게 없으면 LGTM이라고만 답하라"는 출구를 비평가에게 명시적으로 줄 것
  → [26B] 비평이 지목한 파일만 재생성 (전체 재생성 금지 — 콜 절약 + 멀쩡한 파일 보호)
  → [정적+실행] 게이트 재통과 확인. 깨지면 직전 통과 스냅샷으로 rollback
  ↑ 비평-개선 바퀴 기본 1회 (부분합격이면 자동 2회, LGTM이면 조기 종료.
     반복 다듬기는 1/10 비용인 improve 모드가 담당)
  → 최종본 + 비평 히스토리 → 사용자
```

## 역할 분리 원칙 (설계의 핵심 — 바꾸지 말 것)

- **31B = 머리** (설계 + 수용기준 출제 + 비평/채점), **26B = 손** (구현만).
- 수용기준은 31B가 만들고 26B는 따르기만 한다. **출제자·채점자(31B)와 응시자(26B)의 분리**가
  자가채점 후하게 주기(self-grading bias)를 막는 장치다. 26B에게 기준을 만들게 하지 말 것.
- "성공 신호" 커맨드는 실질이 있어야 한다: 단순 exit 0(예: --help만)이 아니라
  아이디어의 핵심 동작 하나를 실제 수행하고 기대 출력이 나오는 것. 수용기준 중 하나를
  실행 가능한 형태로 만들어 성공 신호로 삼는다.
- 설계 산출물은 자연어가 아니라 **구조화(JSON)**로 받는다: 파일 목록(이름+한줄 역할),
  파일별 인터페이스(함수/클래스 시그니처·반환), 의존관계(누가 누굴 import — 이게 0층
  게이트의 기대값이 됨), 핵심사항, 수용기준 초안.

## 가드레일 (헛돌기 방지 — 전부 필수)

- 자가수정은 **층마다 최대 K회 (K=3)**. 초과 시 멈추고 "어디서 막혔는지" 기록 후 종료.
- **진전 없음 감지**: 연속 시도의 diff가 미미하거나 동일 에러 반복이면 K를 안 채우고 즉시 중단.
- **회차당 캡**: API 콜 수·시간·파일 수 상한. 폭주 시 자동 종료.
- **실행 게이트 건당 타임아웃** (기본 30초) + stdin 차단(DEVNULL). 생성물이 입력을
  기다리거나 무한루프에 빠져도 게이트가 멈추지 않게.
- **스냅샷 + rollback**: 게이트를 통과할 때마다 작업 디렉토리를 스냅샷(git commit 권장).
  비평 반영본이 게이트를 깨면 직전 통과본으로 복원.
- 비평가의 무한 트집 방지: LGTM 출구 명시 + 비평 바퀴 고정 상한(기본 1, 부분합격 시 2).

## 정적 검사 원칙

- 정적 검사(`ast` 표준 라이브러리)는 **공짜(콜 0)**다. 모델을 부르기 전에 정적으로 거를 수
  있는 건 다 거른다. 모델 판단은 정적으로 못 잡는 것(의미·의도)에만 쓴다.
- 정적 게이트는 **명백한 것만** 잡는다(파싱 실패, 완전 빈 stub, 대놓고 안 쓰는 import).
  애매한 것(동적 import, getattr 등)은 통과시켜 실행 게이트로 넘긴다 — 과하게 빡빡하면
  멀쩡한 코드를 막는다.
- 자가수정 프롬프트에는 정적 검사가 짚은 **정확한 줄·에러 종류**를 넣는다(한 방 수정률 상승,
  콜 절약).
- 비평가 호출 직전, 정적 검사 결과 요약을 비평 입력에 첨부한다(추측 비평 → 증거 비평).

  가족(26B/31B)만. (상위모델은 이미 계층 오케스트레이션으로 그렇게 쓰여 novelty 0.)

## 행동 규칙 (일하는 방식)

- **구현 중에는 코드 설명·중계를 하지 말 것.** 묵묵히 구현한다.
- 입을 여는 건 딱 두 경우: **(1) 사용자의 결정이 필요할 때** (갈림길, 되돌리기 어려운 선택),
  **(2) 한 단계가 끝났을 때.** 그 외에는 보고하지 않는다.
- 단, 결정을 구할 때는 사용자가 고를 수 있을 만큼의 맥락을 *짧게* 붙인다(선택지만 던지지 말 것).
- 보고는 짧게: "뭘 했나 / 됐나 안 됐나 / 다음에 뭘 할지(또는 뭘 정해줘야 하는지)"만. 코드 나열 금지.
- **큰 결정이나 여러 파일을 한 번에 만들기 전에 먼저 확인받을 것.** 혼자 폭주하지 말 것.
- 한 번에 한 단계씩. 단계가 끝나면 멈추고 짧게 보고한 뒤 다음으로.

## 핸드오프 규칙 (헛돌기 방지 — 필수)

- **새 세션은 HANDOFF.md만 먼저 읽고 바로 시작할 수 있어야 한다.** 다른 문서는 필요할 때만.
- HANDOFF.md **최상단에 `▶ 새 세션 여기부터` 블록을 항상 유지**한다. 그 블록은 셋을 담는다:
  ① 읽는 순서(뭘 먼저) ② **지금 할 일 한 줄**(바로 실행 가능한 단일 액션) ③ 문서 용도표.
- **세션을 끝낼 때**: `지금 어디`·`다음 액션`을 갱신하고, 진입 블록의 "지금 할 일 한 줄"을
  다음 액션 1번과 일치시킨다. 돌던 작업·산출물·다음 시작점을 명시(다음 세션이 재유도 없이 잇게).
- 문서 역할 분리: HANDOFF=현재상태/다음할일, CLAUDE.md=규칙, context-notes=결정로그(왜),
  PLAN=큰명세, game/PAMPHLET=게임규칙. 같은 내용을 여러 곳에 중복하지 말 것.
